Navigation

Навигация в webClient-приложении полностью клиентская. Обращение на сервер для разметкой происходит один раз (или вообще не происходит, в случае offline mode). Все последующие обращения на сервер производятся только как вызове REST-сервисов для получения/изменения данных.

Клиентская навигация связана с History API - новым API в составе HTML5. В тех браузерах, где отсутствует History API (IE<10), клиентская навигация основывается на хеш-url'ах. Для абстрагирования от поддержки History API в браузерах используется библиотека history.js.

При изменении внутри приложения состояния, оно может захотеть отразить его в url'e. Изменять url при изменении состояния приложения необходимо чтобы:

  • работал copy-paste url'a в другой браузере/окно
  • такое поведение является общепринятым для Web- сайтов и приложений
  • приложение должно поддерживать адресацию своих частей и компонентов; это может использоваться для отправки ссылок в письмах и т.п.;
  • был бы возможным возврат на предыдущий шаг по кнопке Назад браузера

Итого: AppState и текущий URL должны соответствовать друг другу. Но они соответствуют не полностью. AppState хранит больше информации, чем текущий Url.

Участники

AppStateManager

Варианты использования AppStateManager:

  • События statechange window
    • обработчик события получает объект текущего состояния используя History.getState и вызывает _applyState
    • метод _applyState в случае, если объект состояния (полученный из History.getState) не содержит прикладных данных (поле data), восстанавливает объект AppState из текущего url'a.
    • метод _applyState вызывает _applyAppState, передавая в него AppState (полученный из состояния History или восстановленный из url)
    • метод _applyAppState:
      • активирует арию из AppState.area (используя AreaManager.activateArea)
      • активирует состояние арии из AppState.areaState, либо используя дефолтное состояние, если имя состояние не задано в объекте AppState (используя Area.activateState)
      • если _applyState/_applyAppState вызывается из обработчика события statechange, либо при старте приложения, то в Area.activateState передается опция disablePushState
  • Application.run вызывает метод AppStateManager.start
    • start получает текущее состояние используя History.getState
    • start выполяет нормализацию url'a: html4-url в html5-браузере меняется на html5-url, html5-url в html4-браузере меняется на html4-url
    • в html5-браузере или в html4-браузере для url'а без хеша ("#") вызывается _applyState (в другом случае сработает событие 'statechange')
  • Вызван метод applyState - вызывается метод _applyState с опциями disablePushState: false
  • Вызван метод switchState - вызывается метод _applyState с опциями fullSwitch: true, disablePushState: false

Area

Варианты использования Area:

  • AppStateManager вызвал метод activateState

Region

  • activatePart принимает объект парта и опции (options)
    • если переданный парт уже активен и он поддерживает метод changeState, то регион вызывает его с объектом из options.state.regionState.partOptions, в любом случае выход
    • если переданный парт существует в списке партов региона (но не активный), то выгружаются (с reason="close") все парты в списке партов региона между найденным и текущий активным
    • если в регионе есть активный парт, то он сначала выгружается (с reason="unload"):
      • для выгрузки парта сначала вызывается его метод queryUnload, который может вернуть
        • строку - выгрузка запрещена, строка используется как причина (reasonToStay)
        • объект с полем reasonToStay - выгрузка запрещена
        • Deferred - объект, на котором будет ожидать регион, может быть resolved/rejected в один из предыдущих результатов
      • если queryUnload разрешил выгрузку, то после него вызывается метод unload парта
      • при выгрузке парта его DOM-элемент скрывается и удаляется из документа
      • после unload парта может быть вызван dispose:
        • если парт входит в активность (Activity) (т.е. при его активации в опциях была задано поле activity - вызов происходил из NavigationService) и причина выгрузки отличная от "suspend", "close", "leave", то активность dispose-ится (т.е. вызывается dispose для каждого парта, входящего в нее)
        • если парт был создан с опцией keepAlive (либо она явно задана для метода _unloadPart), то ничего не делаем
        • иначе если парт был создан с опцией keepStandalone ((либо она явно задана для метода _unloadPart)), то удаляем парт из списка партов региона (не вызывая dispose)
        • иначе удаляем парт из списка партов региона и вызываем для него dispose
    • переданный парт помещается в конец списка партов региона, для него создается обертка PartHelper, которая хранит опции (переданные в activatePart); объект парт присваивается идентификатор в поле __uid
    • регион подписывается на событие statechange парта, обработчик которого перегенерирует его как событие региона statechange
    • у переданного парт вызывается метод render и его DOM-элемент делается видимым
    • после окончания render (render парта может вернуть Deferred), если текущий регион навигируемый (navigable=true), регион генерирует свое событие statechange

Chaos

При активации парта в регионе, вызванной восстановлением состояния AppState, в случае, если парт уже активен, ему вызывается changeState. Проблема: когда мы делаем pushState, то мы фактически устанавливаем состояние приложение, что в себя включает:

  • изменение Url'a
  • изменение заголовка (опционально)
  • добавление записи в историю браузера с предыдущим состоянием

Важно, что в дальнейшем по данному url может быть прямой переход (скопировали url и вставили в новом окне браузера). Т.е. на каждое состояние, созданное с помощью pushState может быть прямой переход.

Сценарии

  • старт приложения варианты:

    • "холодный" старт (новое окно браузера/другой браузер)
    • "горячий" старт (обновление страницы по F5)
    • старт с корнем приложения ("http://localhost/myapp" и "http://localhost/myapp/")
    • старт со сложным url'ом: http://localhost/myapp/display/area1/state1/?param1=value1 обработка в AppStateManager:
    • вызывается start (его вызывает Application.run), в котором производится нормализация url'a для случаев HTML4<->HTML5
    • для HTML5 браузеров вызывается _applyState
    • для HTML4 браузеров ждем события window "statechange" (его генерирует History.js), в обработчике которого вызывается _applyState
    • в _applyState, если объект состояния отсутствует ("холодный" старт), то восстанавливаем его из url (метод _parseRoute)
    • _applyAppState применяет состояние приложения (объект AppState), используя Api AreaManager и Area

  • переход между различными Area и их состояниями

  • активация парта, поддерживающего состояние Например мы находим в состоянии "state1" некоторой area. И хотим активировать новый парт в регионе арии. Для этого, имея ссылку на регион, мы можем явно сказать:

var region = area.regionManager.getRegion("main");
var part = core.createPart("MyPart", {param1:value1});
region.activatePart(part);

При активации парта в регионе с установленным свойством navigable=true состояние приложение должно измениться. Например, url приложения был "/myapp/area1/state1" (или просто "/myapp", если area1 - это дефолтная ария, а "state1" это ее дефолтное состояние). После активации парта url должен измениться на "/myapp/area1/state1/MyPart". Однако

  • изменение своего состояния партом

  • открытие цепочки редакторов и возврат
