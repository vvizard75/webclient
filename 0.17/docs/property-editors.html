Property Editors

Property Editor (pe) - компонент, предназначенный для редактирования значения свойства (или свойств) объекта в редакторе. Базовый класс для всех редакторов свойств - PropertyEditor (lib/ui/pe/PropertyEditor.js).

Создание/метаданные

Для всех редакторов свойств поддерживается общий набор опций. Опции передаются как json-объект в конструктор. Поддерживаемые опции заданы в статическом поле options всех классов редакторов свойств.

name String Наименование свойства
descr String Описание свойств
vt String Наименование типа свойств
hint String Подсказка для интерфейса
nullable Boolean Признак нулабельности свойства
readOnly Boolean Признак свойства только для чтения (по умолчанию false)

#### Опции по умолчанию в зависимости от контекста

В прототипе базового класса редактора свойства PropertyEditor, а так же в прототипах его наследников объявлено свойство `defaultOptions`, представляющее собой объект с набором опций по умолчанию. Например, в NavigationPropertyEditor:

defaultOptions: { selectOptions: { name: undefined, ... openInDialog: false }, ... },

Порой возникает необходимость изменять поведение редактора свойства в зависимости от контекста использования. Например, в фильтре для peObject логичнее открывать ObjectSelector в модальном окне, а не выгружать весь список. Для этого в прототипе класса редактора свойства объявляется свойство `contextDefaultOptions`, представляющее собой словарь, в котором ключ - это имя контекста, а значение - объект переопределяющий опции по умолчанию.  Например, в `NavigationPropertyEditor`:

contextDefaultOptions: { filter: { // в контексте фильтра ({{ObjectFilter}}) селектор открывается в модальном окне. selectOptions: { openInDialog: true } } }

Имя контекста задаётся в свойстве `contextName` прототипов классов `ObjectEditor` и `ObjectFilter`. Для редактора объявлен контекст `"editor"`, для фильтра - `"filter"`.
Для того, что бы подключить механизм переопределения опций по умолчанию, в конструкторе класса редактора свойства в момент построения опций необходимо вызвать метод `PropertyEditor.mixOptions` с указанием контекстных опций по умолчанию:

that.options = that.mixOptions(options, NavigationPropertyEditor.prototype.defaultOptions, NavigationPropertyEditor.prototype.contextDefaultOptions );



> В настоящее время данный механизм реализован в NavigationPropertyEditor (т.е. peObject и peObjectList)



#### Выбор редактора свойства по умолчанию

При определении редактора ([ObjectEditor](objecteditor.html)) возможно явное задание классов редакторов свойств (с помощью поля PropertyEditor метаописания). Однако, обычно задаются только наименования и/или метаданные свойств, без реализаций. В этом случае реализация редактора свойство выбирается автоматически. Для выбора реализации редактора свойства используется статический метод PropertyEditor.DefaultMapping.create. Он принимает метаописание редактора свойства и viewmodel.

Для получения реализации редактора свойства (внутри PropertyEditor.create) используется меппинг, который заполняется с помощью методов PropertyEditor.DefaultMapping. Объект PropertyEditor.DefaultMapping предоставляет API для регистрации отображения свойства на реализацию. Существует две возможности:
1. декларативное отображение типа свойства на реализацию - DefaultMapping рассматривается как словарь отобращения типа свойства на реализацию:

```js
core.ui.PropertyEditor.DefaultMapping.dateTime = peDateTime;
    core.ui.PropertyEditor.DefaultMapping.date = peDateTime;
    core.ui.PropertyEditor.DefaultMapping.time = peDateTime;
  1. регистрация колбека, который будет вызнав при запросе реализации на основе метаданных свойства, колбек должен вернуть класс редактора свойства или null:
core.ui.PropertyEditor.DefaultMapping.register(function(propMd) {
});

В метод DefaultMapping.register также передается объект опций, который может содержать:

  • поле priority - это число, используемое для управления порядком вызова колбэков. По умолчанию priority равно 0, чем больше значение, тем приоритетней
  • поле vt - тип свойства, реализации с одним приоритеом для явно заданного типа свойства, имеют больший приоритет без без типа свойства

Алгоритм получения реализации редактор свойства следующий:

  • вызвать все колбэки, зарегистрированные через DefaultMapping.register для текущего типа свойства (для register был задан объект опция с полем vt)
  • вызвать все колбэки, зарегистрированные через DefaultMapping.register без указания типа свойства
  • если найдена одна реализации, вернуть ее
  • если найдено две реализации, то выбрать ту, которая зарегистрирована с большим приоритетом
  • если найдено две реализации с одинакомым приоритетом, то выбрать ту, которая заданая явно для типа свойства
  • если не найдено реализации, то поискать реализацию с помощью статического меппинга (как поле DefaultMapping для текущего типа свойства)

Пример меппинга для выбора реализаций редактором свойств:

  • для бинарных свойства во всех контекстах используется peBinary, потом что он зарегистрирован с приоритетом 30:
core.ui.PropertyEditor.DefaultMapping.register(function () {
        return peBinary;
    }, { vt: "binary", priority: 30 });
  • в парте просмотра ObjectViewer (contextName равен "viewer") по умолчанию для всех свойств (кроме binary) используется peViewOnly, потому что он зарегистрирован с приоритетом 20:
core.ui.PropertyEditor.DefaultMapping.register(function(propMd) {
        return (propMd.contextName === "viewer" && !propMd.presentation) ? peViewOnly : null;
    }, { priority: 20 });
  • но если для метаданных навигируемого массивного свойства задать presentation="list", то будет использован peObjectList (во всех контекстах)
  • во всех контектах, если задана опция readOnly, то будет использован peReadOnly, потому что он зарегистрирован с приоритетом 10. При этом для peObjectList и peBinary он не будет использован никогда (т.к. у них больший приоритет)
  • все остальные реализации редактором свойств будут выбираться во всех контекстах кроме "viewer" и, если не задана опция readonly - т.к. остальные редакторы регистрируются с приоритетом по умолчанию (или статическим меппингом)
