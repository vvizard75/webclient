Overview

ObjectEditor (редактор) - компонент предназначенный для редактирования объекта. Данным объектом может быть доменный объект, либо его наследник. В любом случае, редактор считает редактируемый объект ViewModel.

ViewModel

ViewModel может быть задана в опциях при создании:

  • как объект
var editor = new ObjectEditor({viewModel: viewModel});
  • как пара тип/идентификатор - в этом случае объект будет загружен (UnitOfWork.get):
var editor = new ObjectEditor({type: "Department", id: "..."});
  • как тип - в этом случае объект будет создан:
var editor = new ObjectEditor({type: "Department"});

ViewModel может быть задана после создания, с помощью вызова метода setViewModel. Если ViewModel задана при создании, то это также сводится к вызову setViewModel. Если объект, который передан в setViewModel не загружен, то запускается его загрузка, а инициализация редактора откладывается до окончания этой загрузки. По завершению загрузки, либо, если объект был загружен изначально, проверяется наличие методов расширения ViewModel: поле {onSetViewModel в объекте опций, переданном в конструктор ObjectEditor, либо наличие метода объекта редактора onSetViewModel (может быть задан в классе наследнике). При наличие заданного колбэка он вызывается с текущей ViewModel, а его результат (если не пустой) считается новой ViewModel редактора.

Пример расширения ViewModel редактора:

var SurveyEditor = core.lang.Class(core.ui.ObjectEditor, {
    constructor: function() {
        this.eventBinder = new core.lang.EventBinder();

        SurveyEditor.Super.apply(this, arguments);
    },

    onSetViewModel: function(viewModel) {
        var that = this;    // it's ObjectEditor
        // расширим модель с помощью хелпера ViewModelExtender
        that.viewModelExtender = new core.lang.ViewModelExtender(viewModel);
        viewModel = that.viewModelExtender.viewModel;
        // добавим свойство showAutogenerated (начальное значение true), при изменении которого будет вызывать наш колбэк
        that.viewModelExtender.declareProp("showAutogenerated", true, function(sender, value) {
            that.getPageByName("autogenerated").hidden(!value);
        });

        return viewModel;
    },

    dispose: function() {
        this.viewModelExtender.dispose();
        SurveyEditor.Super.prototype.dispose.apply(this, arguments);
    }
});

Важно то, что ObjectEditor поддерживает возврат от прикладного обработчика onSetViewModel Deferred-объекта. Т.е. инициализация viewmodel может быть асинхронной. Пример:

var MyEditor = core.lang.Class(core.ui.ObjectEditor, {
    onSetViewModel: function(viewModel) {
        var that = this,
            defer = core.lang.Deferred();
        if (that.viewModel.isNew) {
            // при создании объекта установим свойство user ссылкой на текущего юзера
            viewModel.uow.getCurrentUser().then(function (user) {
                defer.resolve(user);
            });
            return defer;
        }
        return viewModel;
    },

Презентер

Для отображения UI редактор использует презентер. Класс презентера может быть задан явно в опции Presenter. Если он не задан, то будет использован класс, определенный в статическом поле defaultOptions.Presenter. Именно в это поле устанавливает себя презентер редактора по умолчанию:

core.ui.ObjectEditor.prototype.defaultOptions.Presenter = ObjectEditorPresenter;

Аналогичным образом можно изменить презентер по умолчанию.

ObjectEditor ожидает от своего презентера следующее API:

  • render - отобразить общую структуру редактора (без содержимого страниц)
  • renderErrors - отобразить ошибки валидации
  • activatePage - активировать страницу - отобразить ее содержимое
  • deactivatePage - деактивировать страницы - выгрузить ее HTMLDOM

В методы activatePage/deactivatePage передается объект страницы - EditorPage. Страница редактора (EditorPage) хранит коллекцию редакторов свойств. Для отображения страницы также используется презентер. По умолчанию используется объект-класс в поле EditorPage.Presenter. Презентер по умолчанию поддерживает формирование страницы редактора с помощью Handlebars-шаблона. Шаблон может быть задан в описании страницы. Если шаблон не задан, то используется стандартный (lib/ui/templates/EditorPagePresenter.html), который отображает все редакторы свойств из коллекции editors страницы (EditorPage).

Шаблон страницы может содержать элементы с css классом x-pe-container. Если любой редактор свойства внутри этого элемента получает фокус, то элементу-контейнеру добавляется css класс x-pe-focused. А при клике в любом месте такого элемента фокус устанавливается на первый редактор свойства внутри этого элемента (при этому элементу-контейнеру добавляется css класс x-pe-focused). В стандартном шаблоне каждый редактор свойства располагается внутри элемента-контейнера с css классом x-pe-container.

Для шаблона в качестве ViewModel (т.е. текущего объекта) устанавливается объект EditorPage. Для страниц редактора и ее редакторов свойств в качестве ViewModel устанавливается ViewModel редактора.

Презентер, как и сам редактор может иметь опции. Она задаются в свойстве presenterOptions объекта опций редактора. Общие опции презентеров (ObjectEditorPresenterBase):

Наименование Значение Описание
animatePageActivation true/false Анимация при переключении страниц/шагов редактора/мастера. По умолчанию true

Context и вложенные редакторы

Редактор при создании в опциях ожидает объект editorContext. Если он не задан, то редактор считается корневым. Если контекст задан, то в нем может быть установлено свойство nested: true. В этом случае редактор считается вложенным. Корневой редактор от вложенного отличается следующим:

  • при инициализации корневой редактор создает точку отката в UnitOfWork;
  • при сохранении корневой редактор (метод finish, который вызывает команда SaveAndClose) происходит сохранение изменений в UnitOfWork;
  • при отмене корневой редактор выполняет откат изменений в UnitOfWork

UnitOfWork

Редактор ожидает, что переданная ему viewModel входит в UnitOfWork - т.е. имеет свойство uow. Если в редактор передали тип/идентификатор, то UnitOfWork получается либо из объекта опций, либо, если там она не задана, создается через объект Application (core.Application.current.createUnitOfWork()). UnitOfWork используется редактором для следующих целей:

  • для сохранения изменений в корневом редакторе - вызывается метод save(options)
  • для создания точки отката и возврата к ней в случае отмены (в корневом редакторе) - вызывается методы saveState, rollbackState, acceptState
  • для проверки наличия изменений в queryUnload - вызывается метод hasChanges

Редактор поддерживает два режима сохранения: асинхронное и синхронное (блокирующее). По умолчанию используется асинхронное сохранение: редактор вызывает метод uow.save и после этого navigationService.close, т.е. запускается сохранение на сервер (uow.save вызывает DataFacade.save, а он BackendInterop.save) и текущий редактор закрывается (мы возвращаемся в тот парт, который вызвал navigationService.navigate).

В режиме синхронного сохранения редактор не вызывает закрытия через navigationService.close сразу же после запуска uow.save. Вместо этого он устанавливает свойство isBusy=true, что приводит к блокированию интерфейса, и дожидается завершения операции. В случае успеха изменения в UnitOfWork фиксируются и редактор закрывается. В случае ошибки, она отображается по аналогии с ошибками валидации, интерфейс разблокируется и закрытия не происходит. Синхронный режим включается с помощью опции blockingSave.

Команды и меню

Редактор может содержать (и по умолчанию содержит) меню. Для отображение меню отвечает презентер редактора. По умолчанию используется MenuButtonsPresenter (меню в виде кнопок). Экземпляр меню редактора создается и инициализируется в конструкторе ObjectEditor'a. Для создания меню используются метаданные, которые ищутся в следующем порядке:

  • options.menu
  • Menu.Defaults.Editor

Т.е. если в свойстве menu объекта параметров, переданном в конструктор обнаружен объект, то он используется как метаданные меню. В противном случае, метаданные берутся из статического поля Menu.Defaults.Editor. Поле Menu.Defaults.Editor по умолчанию заполняется в специальном модуле app\default-menus.js, который обычно подключен.

В реальности не важно, где заполняется это поле, главное чтобы оно было. Файл default-menu.js входит в состав дистрибутива, но может меняться - это набор метаданных меню по умолчанию.

После создания меню оно связывается с командами редактора. Т.о. в меню можно не задавать явно команды, если нужны стандартные. Редактор поддерживает следующие стандартный команды:

Команда Описание
SaveAndClose Сохранить и закрыть, вызывает метод finish
CancelAndClose Отменить и закрыть, вызывает метод cencel
SwitchToPage Переключиться на страницу с заданным именем, вызывает метод switchToPage

Меню по умолчанию редактора выглядит как:

Menu.Defaults.Editor = { items: [
        { name: "SaveAndClose" },
        { name: "CancelAndClose" }
    ]};

"SaveAndClose" и "CancelAndClose" здесь - это наименование стандартных команд.

Validation

Overview

Валидация в редакторе всегда выполняется перед сохранением. Дополнительно она может выполняться для текущей страницы при уходе с нее - этим управляет параметр pageValidation, см. #Валидация при уходе со страниц/шагов редактора.

Если требуется полностью отключить валидацию, то для этого следует задать обработчик onFinish.

Правила валидации для редактора

Правила валидации могут быть заданы:

  • для редактора
  • для страницы
  • для редакторов свойств

Для редактора правила валидации задаются в коллекции rules метаданных редактора:

var editor = core.ui.ObjectEditor.create(app, {
    viewModel: viewModel,
    pages:[{
        name: "page name",
        properties: [ "prop1" ]
    }],
    rules: [{
        validate: function(obj) {
            if (obj.prop1() === "42")
                return "Answer to the Ultimate Question of Life, the Universe, and Everything";
        }
    }]
});

Метод validate принимает экземпляр объекта, а возвращает то же самое, что метод validate объекта (строку или объект ошибки) (см. Validation).

Для страницы редактора правила валидации задаются в коллекции rules метаданных страницы:

var editor = core.ui.ObjectEditor.create(app, {
    viewModel: viewModel,
    pages:[{
        name: "page name",
        properties: [ "prop1" ],
        rules: [{
            validate: function(obj) {
                if (obj.prop1() === "42")
                    return "Answer to the Ultimate Question of Life, the Universe, and Everything";
            }
        }]
    }]
});

Сигнатура метода validate аналогична.

Правила валидации для отдельных свойств:

var editor = core.ui.ObjectEditor.create(app, {
    viewModel: viewModel,
    pages:[{
        name: "page name",
        properties:[{
            name: "prop1",
            rules: [{
                validate: function(propValue) {
                    if (propValue === "42")
                        return "Answer to the Ultimate Question of Life, the Universe, and Everything";
                }
            }]
        }]
    }]
});

В этом случае метод validate принимает значение свойства.

Валидация при уходе со страниц/шагов редактора

Поведение валидации при перемещении по закладкам/шагам задаётся атрибутом pageValidation в метаданных редактора. Возможные значения:

  • none - валидация при уходе со страницы/шага не выполняется. Используется по умолчанию;
  • loose - выполняется валидация редакторов свойств для страницы, при наличии нарушения уход не запрещается, соответствующая закладка подсвечивается и на ней появляется признак ошибки. По умолчанию для мастера (pageStyle: "wizard");
  • strict - аналогично loose, но запрещается уход со страницы/шага.

Navigate Siblings

Если в редактор передана опция navigateSiblings с массивом объектов, то по умолчанию в редакторе отображается "карусель" (core.ui.Carousel), позволяющая переходить между объектами - т.е. закрывать текущий редактор и открывать редактор следующего объекта без возврата в список. Элемент массива navigateSiblings используется как объект опций для открываемого парта.

###### Прикладные обработчики
Прикладные обработчики могут быть заданы двумя способами: 
* в объекте options, передаваемом в конструктор
* как методы в объекте ObjectEditor - в самом объекте или в прототипе (т.е. классе, унаследованном от ObjectEditor).

Все обработчики в ObjectEditor вида onXXX генерируют соответствующее событие (обработчик "onXyz" генерирует событие "xyz"). Все события ObjectEditor описаны в объекте статическом поле events.

Колбэки, передаваемые в объекте options в конструктор ObjectEditor, автоматически подписываются на события по соглашению: колбэк в поле "onInitialize" подписывается на событие "initialize" (т.е. имя поля без "on" является наименованием события).

Итого существуют следующие способы подключения прикладной логики в редактор:
* создать класс наследник и в нем определить метод onXyz, не забыть вызывать метод базового класса (если только не требуется предотвратить генерацию события):

```js
var MyEditor = core.lang.Class(ObjectEditor, {
    constructor: function() {
        MyEditor.Super.apply(this);
    },
    onPageStarted: function (args) {
        // my logic
        ObjectEditor.prototype.onPageStarted.call(this, args);
    }
});
var editor = new MyEditor(options);
  • заменить метод onXyz в объекте:
var editor = new MyEditor(options);
editor.onPageStarted = function (args) {
    // my logic
    ObjectEditor.prototype.onPageStarted.call(this, args);
}.bind(editor);
  • подписаться на событие объекта:
var editor = new MyEditor(options);
editor.bind(editor.events.PAGE_STARTED, function (args) {
    // my logic
});
  • передать колбэк в options 
options.onPageStarted = function (args) {
    // my logic (this будет editor)
}
var editor = new MyEditor(options);

Обработчики onXyz в ObjectEditor и события имеют одни и те же аргументы - объект args (набор полей зависит от события).

События, имена которых заканчиваются на "ing" (типа unloading, starting) обычно поддерживают поле cancel, установка в true которого прерывает начатую операцию.

Страницы редактора

Страницы редактора описываются свойством pages в виде массива метаописателей страниц.

var editor = core.ui.ObjectEditor.create(
    {
        title: "Editor Title",
        pages: [
            {
                name: "page1",
                title: "Page 1 Title",
                properties: [
                    "prop1",
                    {
                        name: "prop2"
                        presentation: "radio"
                    }
                ]
            },
            {
                name: "page2",
                title: "Page 2 Title",
                properties: [
                    "prop3", "prop4"
                ]
            }
        ]
    }
);

Каждый объект - метаописатель страницы, может содержать следующие опции:

Наименование Значение Обязательность, значение по умолчанию
title Заголовок необязательно
name Имя страницы необязательно, в случае отсутствия будет задан как "page-номер_страницы"
hidden видимость закладки (см. ниже) false
template handlebars шаблон страницы необязательно
bound привязка данных в шаблоне выполняется кодом? false
labelColumnRatio соотношение ширины столбцов страницы "label"/"property editor" в терминах грида twitter bootstrap 4
cssColumnPrefix Префикс для css-классов, задающих ширину столбцов в гриде twitter bootstrap. Возможные значения: col-xs-, col-sm-, col-md- или col-lg- col-md-
highlightFocused подсвечивать или нет активный редактор свойства true для редактора, false для фильтра
Управление видимостью закладок

При определении редактора для страницы может быть задан параметр hidden: true для начального сокрытия закладки:

editor = new ObjectEditor({
        pages:[{
            name: "pageName1",
            hidden: true
        }]
    });

В рантайме управление видимостью закладки осуществляется путем установки свойства hidden для объекта-описателя страницы:

editor.getPageByName("pageName1").hidden(true);
Управление видимость редакторов свойств

Аналогично закладкам для редакторов свойств может быть задано свойство hidden либо при создании ObjectEditor, либо в runtime:

editor = new ObjectEditor({
        pages:[{
            name: "pageName1",
            properties: [ { name: "prop1", hidden: true } ]
        }]
    });

или

var page = editor.getPageByName("pageName1");
page.getPropertyEditor("prop1").hidden(true);

Кастомный шаблон страницы.

По умолчанию каждая страница редактора использует стандартный шаблон, отображающий все редактора свойств (PE) друг под другом. Стандартный шаблон использует partial шаблон >peContainer. Разметку страницы можно изменить, задав свой шаблон с помощью опции template. При этом также рекомендуется использовать partial шаблон >peContainer и хелперы peContainer и peLayout. Пример:

<form onsubmit="return false" class="my-custom-page">
{{#peLayout columnsRatio="2:10"}}
    {{> peContainer editors.title}}
    {{> peContainer editors.organizer}}
{{/peLayout}}
</form>

Если нужно разместить два и более PE в одну строку, рекомендуется следующий подход:

<form onsubmit="return false" class="my-custom-page">
    {{peContainer editors.title columnsRatio="2:10"}} {{!-- один PE на строке --}}
    <div class="row"> {{!-- строка с двумя PE --}}
        <div class="{{options.cssColumnPrefix}}6">
            {{peContainer editors.createdFrom columnsRatio="4:8"}}
        </div>
        <div class="{{options.cssColumnPrefix}}6">
            {{peContainer editors.createdTo columnsRatio="4:8"}}
        </div>
    </div>
</form>

Как правило, хочется, чтобы первые PE на строке с несколькими PE и однострочные PE были выровнены по левому краю. Чтобы этого достичь, необходимо выполнение следующего равенства: {center}labelRatio для однострочного PE === peContainerRatio * labelRatio / 12 для первого PE на строке{center} В примере выше: однострочный PE - title, первый PE на строке - createdFrom

  • labelRatio для однострочного PE - 2 (т.к. {{columnsRatio="2:10"}})
  • peContainerRatio для первого PE на строке - 6 (т.к. {{class="{{options.cssColumnPrefix} }6"}})
  • labelRatio для первого PE на строке - 4 (т.к. {{columnsRatio="2:10"}})

Подставляем в формулу: 2 === 6 * 4 /12. Таким образом, PE для свойств title и createdFrom будут выровнены по левому краю.
