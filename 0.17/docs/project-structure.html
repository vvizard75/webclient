Project Structure

Структура прикладного проекта

Существует несколько подходов к организации структуры проекта на WebClient.

Single NuGet-based Project

Солюшен VS содержит один ASP.NET проект, в котором установлены NuGet-пакеты для серверной (XFW3.WebClient.Server) и клиентской (XFW3.WebClient.Client) части. Для удобства можно устанавливать один пакет - XFW3.WebClient, который зависит от .Server и .Client. Файлы клиентской части из пакета WebClient.Client при установке будут добавлены в проект VS. Если проект хранится в TFS, то и автоматически в TFS. При каждом обновлении пакета все файлы из пакеты будут удаляться и добавляться заново, это особенность NuGet. Поэтому обновление NuGet-пакета с клиентской частью может быть довольно долгим в проектах, связанных с TFS. К плюсам данной конфигурации относятся:

  • минимум используемых инструментальных средств (VS)
  • рантайм-приложение (виртуальный каталог в IIS) использует текущую версию скриптов, т.о. изменения в них видны сразу - т.е. нет необходимости в "билде" клиентской части

В минусам относятся:

  • сложность обновления клиентской части
  • все файлы клиентской части хранятся в TFS
  • сложность замены или модификации файлов из WebClient
  • объединение клиентской и серверной части может быть неудобным в случае нескольких платформ - например, клиентская часть используется в Chrome Web App, Firefox OS, Cordova-based приложениях и т.п..
  • сложность в получении различных дистрибутивов - например development и production версий
  • для работы с front-end проектами используется свой инструментарий (npm/grunt/yeoman), артефакты которого придется храниться в папке проекта VS

Separated Server/Client Projects

Другой возможной структурой организации проекта является разделение проектов для серверной и клиентской части приложения. Серверная часть - это ASP.NET VS-проект, в котором установлен NuGet-пакет WebClient.Server (и все его зависимости - Xfw3, ASP.NET MVC/WebApi и т.д.). Клиентская часть - отдельный каталог, в котором установлены NPM-пакеты, необходимые для разработки. В качестве IDE может (и рекомендуется) использоваться JetBrains WebStorm.

Именно таким образом организована разработка самого WebClient.

Установка клиентской часть в данном случае выполняется с помощью bower - см. Distribution.

Getting Started

Перед началом убедитесь, что выполнены все шаги из Setup Developer Environment.

Теперь мы готовы создать проект. Это можно сделать вручную, либо воспользоваться генератором для yeoman - сначала его надо установить:

  • установить yeoman:
npm install yo -g
  • установить yeoman-генератор для WebClient из приватного репозитория:
npm install -g generator-croc-webclient --registry=http://rnddev01.lab.croc.ru:8006
  • запустить генератор в папке для клиентского проекта (e.g. "client") :
yo croc-webclient:client

Генератор задаст несколько вопрос и создать рекомендуемую структуру проекта. После этого автоматически запустится установка зависимостей (если не задан флаг --skip-install) - всех npm-пакетов и bower-пакетов. В виде npm-пакетов устанавливаются различные инструменты, используемые во время разработки. В виде bower-пакета устанавливается дистрибутив клиентской части WebClient.

Запускать генератор надо один раз, чтобы создать первоначальную структуру проекта.

В папке Server содержится VS-проект. Т.к. сайт/приложение IIS'а обычно указывает на папку проекта, то здесь же должна содержаться клиентская часть. Серверной части надо знать где находится клиентская часть в Web-приложении, чтобы сформировать корректные ссылки на скрипты для клиента. По умолчанию это папка "client", но ее наименование может быть задано в appSettings в web.config в ключе "croc.clientBase". Файлы клиентской части в Server/client напрямую не правятся, они копируются туда тасками Grunt. Grunt - это task runner на основе node.js для front-end проектов. В папке Client должны быть:

  • .bowerrc - конфигурация bower, здесь задан url приватного репозитория для установки дистрибутива WebClient
  • bower.json - bower-конфигурация проекта, здесь указаны пакеты от которых зависит проект (поле dependencies), но вы также можете устанавливать любые другие пакеты, это удобный способ управления зависимостями для front-end проекта

  • package.json - конфигурация проекта для npm, т.к. сам проект клиентской части не публикуется как npm-модуль, то фактически из package.json используется только поле devDependencies - это набор npm-пакетов, от которых зависит проект;

    Т.к. часть пакетов могут содержать только в нашем приватном репозитории, то для их установки надо задать параметр registry: 

    npm install --registry=http://rnddev01.lab.croc.ru:8006

  • Gruntfile.js - сценарий для Grunt'a с описанием тасков

С помощью тасков Grunt могут выполняться следующие операции:

  • формирование dev-дистрибутива (as is)
  • формирование production-дистрибутива (выполнение оптимизации и минификации)
  • копирование дистрибутива в серверный проект
  • очистка папки client в сервером проекте
  • запуск фонового процесса для слежения за файлами клиентской части для автоматического копирования их в серверный проект
  • выполнение юнит-тестов
  • запуск jshist/jslink и т.п.
  • множество других - см. доступные плагины для Grunt - http://gruntjs.com/plugins

При создании проекта с помощью Yeoman-генератора в проекте создается Gruntfile.js с набором необходимых типовых тасков:

  • grunt dev - копировать WebClient и прикладной код в серверный проект
  • grunt watch - запуск фонового процесса отслеживания модификации файлов
  • grunt build - формирование дистрибутива (в папке dist/dev)
  • grunt build:dist - формирование production-дистрибутива (в папке dist/prod)
  • grunt build:dev - формирование dev-дистрибутива и копирование его в серверный проект
  • grunt build:prod - формирование production-дистрибутива и копирование его в серверный проект

При создании клиентского проекта с помощью Yeoman-генератора генератор спрашивает относительный путь до серверного проекта - как раз для того, чтобы его прописать в тасках в Gruntfile.js

Для формирования production-дистрибутива используется Npm-пакет grunt-croc-requirejs.
