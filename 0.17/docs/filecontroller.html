FileController

{toc}

Overview

Функциональность по работе с бинарными данными выделена в модуль XFW3.WebClient.Files. При добавлении сборки в bin каталог веб-приложения, модуль автоматически регистрируется в инфраструктуре WebClinet. Сборка входит в NuGet-пакет XFW3.WebClient, поэтому устанавливается автоматически. Но не является обязательной.

Сборка XFW3.WebClient.Files содержит следующие компоненты:

  • FileControllerBase - абстрактный класс для реализации FileController; Ожидается, что класс FileController будет создан в прикладном проекте. В простейшем случае он может просто наследовать от FileControllerBase. При установке NuGet-пакета WebClient, в прикладном проекте автоматически создается файл FileController.cs с таким классом;
  • ModuleBootstrapper - код инициализации модуля, подключаемый с помощью атрибута PreApplicationStartMethod, регистрирует все необходимые компоненты в инфраструктуре WebClient;
  • FilesModuleConfig - конфигурация модуля (будет доступна в коллекции XConfig.Modules по ключу "files");
  • FilesRoutesConfig - конфигурация маршрутов ;
  • XFileStorage - реализация интерфейса IXFileStorage, с помощью которой функциональность загрузки файлов интегрируется с общим механизмом сохранения/загрузки WebClient'a (DomainApiController)

Инициализация

В процессе инициализации модуля в Ninject-контейнере регистрируется реализация IXFileStorage, если она там отсутствует, - XFileStorage.

Работа с файлами

XFileStorage разделяет каталоги для загрузки и для временного хранения. Под основным каталогом Files содержаться еще два: Inbound и Uploaded. В каждом из них создаются подкаталоги для пользователей. В Inbound файлы загружаются контроллером FileControllerBase (путь до этого каталога возвращает IXFileStorage.GetUserInboundDir). Когда пользователь загружает файл, используя API Upload, то файл создается в Inbound. После того как файл загружен, он регистрируется в FileStorage (с помощью метода IXFileStorage.RegisterFile). В этот момент FileStorage переносит файл из Inbound в Uploaded. При инициализации XFileStorage:

  • полностью очищает папку Inbound.
  • считывает существующие файлы из Uploaded. Для файлов он также считывает время создание файла и его размер. При инициализации из Uploaded удаляются файлы старее периода, заданного в конфигурации модуля FilesModuleConfig.FilesMaxAge (по умолчанию 2 дня).

Перед началом загрузки файла FileController и FileStorage проверяют квоту на общий размер загруженных файлов текущим юзером. Для этого в конфигурации модуля (FilesModuleConfig) есть параметры для задания квот:

  • QuotaPerUser - максимальный сумарный размер файлов в XFileStorage для одного пользователя (суммарно в Inbound и в Uploaded)
  • QuotaTotal - максимальный сумарный размер файлов для всего приложения. Перед началом загрузки также учитывается размер свободного места на диске (если размер файла превышает размер свободного места, то загрузка прерывается).

В случае прерывания загрузки (по любой причине), временный файл в Inbound удаляется.

Файл из Uploaded удаляется в случае успешного сохранения объекта, который содержит ссылки на загруженные ресурсы. Это означает, что файлы в каталоге Uploaded могут накапливаться (загрузили файл и закрыли браузер).

Конфигурация

Класс FilesModuleConfig содержит конфигурацию модуля, которая может быть задана либо в рантайме, либо декларативно в main.json.config. Свойства:

Свойство Значение Значение по умолчанию
RootDirectory Абсолютный или относительный каталог для хранения файлов FileController'ом. Если не задан, то используется папка, наименование которой задано в свойстве XFileStorage.RootDirName, относительно каталога web-приложения. Files
FilesMaxAge Максимальный период хранения загруженных файлов. 2 дня
QuotaPerUser Квота на размер загруженных файлов (в сумме) для одного пользователя. 1 Гб
QuotaTotal Квота на общий размер загруженных файлов для всех пользователей. Int64.MaxValue
UploadChunkSize Размер чанка (добавлено в версии xfwajax:Release 0.17) Вычисляется в рантайме при старте как наименьшее из ограничений

Маршруты

Базовый Uri всех маршрутов: "api/_file/" (задается в FilesModuleConfig).

Uri Method Обработчик
/binaryPropValue GET FileControllerBase.GetBinaryPropValue
/resource GET FileControllerBase.GetResource
/resource/delete POST FileControllerBase.DeleteResource
/Upload POST FileControllerBase.Upload

FileController

Api-контроллер FileController используется для загрузки и отображения Binary-свойств доменных объектов. Загрузка осуществляется с помощью метода Upload. Он ожидает POST-запрос с ContentType multipart/form-data запрос. Данные из http-запроса читаются с помощью стандартного компонента из WebApi - MultipartFormDataStreamProvider. Данные из http-запроса сохраняются в файл в директорию, путь до которой формирует XFileStorage. По умолчанию файл загружается в папку с именем текущего пользователя (получаемого от XSecuritySubsystem), которая содержится в подпапке Inbound. Для анонимных пользователей и/или отсутствии XSecuritySubsystem в приложении файлы загружаются в подпапку "All Users".

По умолчанию FileControllerBase не разрешает загружать файлы для пользователей (кроме случая отсутствия XSecuritySubsystem в приложении). Для проверки возможности загрузки контроллер проверяет наличие привилегии "FileUpload" у текущего пользователя. Это выполняется в методе FileControllerBase.IsUploadAllowed и может быть переопределено.

Привилегии для пользователей устанавливаются в UserProvider'e.

В качестве результата загрузки файла FileControllerBase возвращает json-объект с полем resourceId. Данный resourceId используется для ссылки на загруженный ресурс. Используя resourceId можно скачать и удалить файл (но только свой). Для этого используются методы GetResource и DeleteResource контроллера.

На клиенте resourceId помещается в описатель значения Binary-свойства и передается при сохранении объекта на сервер. На сервере при сохранении объекта для Binary-свойства, содержащего resourceId, получаются данные файла с помощью IXFileStorage.GetFileStream и помещаются в Binary-свойство доменного объекта. Файл ресурса при этом удаляется.

Большие файлы

По умолчания IIS и ASP.NET ограничивают максимальный размер запроса. Ограничение ASP.NET задается в system.web/httpRuntime/@maxRequestLength в киллобайтах. Ограничение IIS задается в system.webServer/security/requestFiltering/requestLimits/@maxAllowedContentLength в байтах.

Ограничение Тип Значение по умолчанию Максимальное значение
IIS (maxAllowedContentLength) UInt32 30000000 (bytes, ~28.6MB) 4294967295 (4Gb)
ASP.NET (maxRequestLength) Int32 4096 (KB, 4Mb) 2147483647 (2Tb)

Таким образом теоретический максимальный размер файла, который может быть загружен, 4Gb. Однако, по факту файлы больше 2Gb не загружаются - при отправле файлов большего размера IIS возвращает ошибку 400 Bad Request, даже не вызывая контроллер.

Для обхода ограничений на максимальный размер запроса используется кусочная загрузка. В конфигурации FilesModuleConfig может быть задано свойство UploadChunkSize, которое определяет размер chunk'a. Свойство возвращается на клиент как опция модуля files и используется в peBinary для задания опции maxChunkSize плагина jquery.fileupload. Именно этот плагин реализуется кусочную загрузка. Однако, для "разрезания" файла на куски, он использует Blob API, которое поддерживается не всеми браузерами, конретно следующими:

  • Desktop browsers
    • Google Chrome
    • Apple Safari 6.0+
    • Mozilla Firefox 4.0+
    • Opera 12.0+ (broken, Opera fails to properly upload Blob objects)
    • Microsoft Internet Explorer 10.0+
  • Mobile browsers
    • Apple Safari Mobile on iOS 6.0+
    • Google Chrome on iOS 6.0+
    • Google Chrome on Android 4.0+

Если свойство FilesModuleConfig.UploadChunkSize явно не задано, то при инициализации сервера вычисляется наименьшее из значений maxAllowedContentLength/maxRequestLength в конфигурации и устанавливается в качестве UploadChunkSize.
