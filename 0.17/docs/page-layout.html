Page Layout

Приложение на базе WebClient - это Single Page Application (SPA). Для навигации в приложении используется клиентская навигации (на базе History API или hash-фрагментов). Загрузка страницы с сервера выполняется только при первом открытии приложения и при явном обновлении страницы браузера.

Загрузка всех URL'ов с клиента на сервере контролируется конфигурацией маршрутов. Конфигурация маршрутов (отображение паттернов URL'ов на обработчики) выполняется в RoutesConfig. Подробней см. Server Architecture. По умолчанию все маршруты "страниц" ведут в один MVC контроллер XHomeController. Стандартный модуль RoutesConfig выполняет настройку следующих MVC-маршрутов (помимо представления есть еще маршруты для выполнения REST сервисов):

routes.MapRoute(
            name: "Display",
            url: "display/{*pathInfo}",
            defaults: new
            {
                controller = "XHome",
                action = "Index"
            }
        );
        routes.MapRoute(
            name: "Home",
            url: "",
            defaults: new
            {
                controller = "XHome",
                action = "Index"
            }
        );

Два маршрута: корень приложения и все URL'ы, начинающиеся с "/display/", ведут в один метод Index контроллера XHomeController. Данный метод (Index) возвращает View под названием "Index". Данное View должно присутствовать в прикладном приложении:

В состав WebClient не входит cshtml-шаблоны представления (view), т.к. они принадлежат конечному приложению. Однако, к возвращаемому html-коду из Home-контроллера предъявляются определенные требования, рассмотренные ниже.

Для удобства типовые элементы разметки оформлены как отдельные cshtml-файлы, начинающиеся на "_Import". В типовой реализации _Layout.cshtml эти файлы импортируются с помощью вызова @RenderPage. К таким типовым элементам относится:

  • _ImportCssLinks.cshtml - множество ссылок (link) на css файлы
  • _ImportJavascriptInline.cshtml - вставка глобальных js-объектов
  • _ImportJavascriptLinks.cshtml - вставка ссылок на js-скрипты

Для получения базовой реализации MVC view "Index" можно воспользоваться NuGet-пакетом WebClient.Sample.

глобальные объекты конфигурации

Разметка должна содержать тег script с определением двух глобальных объектов require и xconfig. Объект require - это конфигурация для библиотеки require.js, на которой основан WebClient. Объект xconfig - это конфигурация приложения, полученная путем объединения json-конфигурации из файла main.config.json и глобального .net-объекта XConfig (фактически возвращаемый клиенту json-код содержится в свойстве XConfig.ClientConfig).

Home-контроллер во вью передает объект модели MainPageModel, которая содержит js-код для определения данных объектов (свойства ClientConfigJsonString и RequireConfigJsonString). Модель MainPageModel инициализируется методом PageControllerBase.initializePageModel.

Для вставки необходимого кода можно воспользоваться _ImportJavascriptInline.cshtml:

@RenderPage("~/Views/Shared/_ImportJavascriptInline.cshtml")

См. также main.js.

включение css-стилей

Разметка должна содержать ссылки на "корневые" css-скрипты:

  • "root.css" из lib
  • generic.css
  • "root.css" из app

Для вставки необходимого кода можно воспользоваться _ImportCssLinks.cshtml:

@RenderPage("~/Views/Shared/_ImportCssLinks.cshtml")
включение js-скриптов

Возвращаемая разметка должна содержать js-скрипты:

  • ссылку на main.js
  • ссылки на shim-скрипты
  • ссылки на require.js

В версии 0.11 произошли существенные изменения - появился механизм Bootloader

Для вставки необходимого кода можно воспользоваться _ImportJavascriptLinks.cshtml:

@RenderPage("~/Views/Shared/_ImportJavascriptLinks.cshtml")

Данный инклуд вызывает extension-метод WriteBootScripts, который:

  • добавляет пустой тег SCRIPT с атрибутом data-main, содержащим ссылку на main.js (берется из свойства MainPageModel.MainScriptServerPath), данный тег будет использоваться require.js для начала загрузки всех скриптов
  • запрашивает у Bootloader'a список boot-скриптов (скрипты из папки boot) и добавляет из в тег SCRIPT; по умолчанию это будет один скрипт - bootloader.js

Связка Bootloader/BootloaderController/bootloader.js используется для формирования списка начальных скриптов на основании параметров клиента.

Механизм Bootloader'а опциональный. При необходимости его можно игнорировать и формировать ссылки на js-скрипты сразу в возвращаемой разметке

Клиентский скрипт bootloader.js делает две вещи:

  • вставляет новый тег SCRIPT с атрибутом src, указывающим на url "bootloader", передавая в него параметры. На текущий момент bootloader.js передает два параметра: device=mobile для тач-устройств и screen=width:height, где width и height - это ширина и высота экрана соответственно.
  • находит на странице элемент с идентификатором _appcacheframe и устанавливает ему атрибут src в зачение из атрибута data-src, дополняя его теми же параметрами - это необходимо для поддержки AppCache

Url "bootloader", на который ссылается динамически вставленный тег SCRIPT, ведет к контроллеру BootloaderController (маршрут создается в RoutesConfig). В качестве результата контроллер должен вернуть js-код. Это код получается путем объединения js-скриптов, заданных в конфигурации bootloader'a в main.config.json с учетом параметров клиента (параметром переданных в query string из bootloader.js и userAgent). Идея bootloader'a в том, чтобы вернуть набор скриптов с учетом возможностей текущего клиента. До версии 0.11 список скриптов, который возвращал сервер, был захардкожен в _ImportJavascriptLinks.cshtml.

Подробней см. в Bootloader.

Типовая реализация "Index" View

View "Index" типизированное - оно получает модель типа MainPageModel. Модель содержит необходимые данные для формирования представления. Для View Index.cshtml используется Master Layout. Он может быть задан либо явно в Index.cshtml, либо (рекомендованный подход), ссылка на него задается в файле _ViewStart.cshtml: 

@{
    Layout = "~/Views/Shared/_Layout.cshtml";
}

Таким образом для всех View задается Master Layout - _Layout.cshtml.
