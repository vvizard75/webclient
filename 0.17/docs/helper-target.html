Helper target

Для работы некоторых хелперов требуется DOM-элемент. К числу таких хелперов относятся:

  • data-bind
  • command-bind
  • render
  • observe
  • unbound
  • part
  • pe

Первым двум хелперам DOM-элемент необходим, чтобы создать binding к нему; остальным - чтобы отрендерить что-то в него. Однако, Handlebars-шаблоны генерируют HTML в виде строки, а DOM-элементы создаются из этой строки позднее. Чтобы обойти это ограничение вышеперечисленные хелперы на этапе обработки шаблона добавляют ко View специальные callback-и, которые вызываются после создания DOM-элементов. В качестве одного из аргументов callback-а хелпер передает некоторую информацию о целевом DOM-элементе. В простейшем случае это ID нужного элемента. Но откуда он берется? Рассмотрим на примере data-bind. Вот типичное использование этого хелпера:

<input type="text" {{data-bind value="lastName"}} />

Как видно, никакого ID здесь не фигурирует. На самом деле хелпер data-bind сам генерирует новый уникальный ID (с помощью метода View.newId()) и подставляет его в текст HTML. Т.е. на выходе получим такой HTML:

<input type="text" id="x17" />

Значение атрибута id может отличаться, главное - что он уникален. Этот id хелпер и передает во View как аргумент колбэка. В этом колбэке будет найден нужный элемент и выполнен data binding к нему (все это происходит уже после обработки шаблона). Аналогично работает хелпер command-bind. Эти два хелпера просто подставляют в текст шаблона атрибут id, поэтому должны размещаться в теге какого-то элемента на ряду с другими атрибутами. Остальные хелперы более сложны: они могут сами создавать нужные им элементы. Для примера рассмотрим хелпер render, все сказанное полностью аналогично и для других хелперов. Вот простейший вариант использования хелпера:

<div>{{render innerPart}}</div>

В результате обработки этого шаблона будет сгенерирован примерно следующий HTML:

<div><span id="x21"></span></div>

Хелпер сам создает теги для своего целевого элемента и присваивает ему новый ID (он может отличаться от приведенного в примере). На этапе обработке колбеков этот элемент будет найден и в него отрендерится парт, который является значением свойства innerPart. При этом внутрь сгенерированного span скорее всего добавятся еще какие-то элементы, это зависит от реализации метода render в парте. Генерируемый хелпером HTML можно немного кастомизировать. Если нужно использовать не span, а другой тег, его можно задать в атрибуте tag хелпера:

<div>{{render innerPart tag="div"}}</div>

Также к сгенерированному элементу можно добавить css-класс с помощью атрибута cssClass:

<div>{{render innerPart tag="div" cssClass="inner-part"}}</div>

Результатом будет примерно следующее:

<div><div id="x21" class="inner-part"></div></div>

Указание целевого элемента для хелпера.

Возможности, описанные в данном разделе, добавлены в версии 0.10.

Как можно заметить, возможностей для кастомизации HTML совсем немного - всего два атрибута tag и cssClass. Что делать, если нужно добавить еще какие-то атрибуты, например, title или data-*? Что делать, если нужно явно задать предопределенный ID? В конце концов, зачем создавать вложенный элемент, если уже есть элемент-контейнер? Чтобы ответить на все эти вопросы в версии 0.10 для хелперов был добавлен атрибут target. Значением атрибута является css-селектор целевого элемента. Пример:

<div class="inner-part" title="Вложенный парт" data-state="init">{{render innerPart target=".inner-part"}}</div>

Результатом такого шаблона будет просто:

<div class="inner-part" title="Вложенный парт" data-state="init"></div>

Т.е. в этом случае хелпер не генерирует никакого дополнительного HTML. Он лишь запоминает селектор, указанный в target, и передает его в колбэк View. В колбэке нужный элемент ищется не по ID, а по переданному селектору. Более того, так как дополнительный HTML не создается, то размещать хелпер с атрибутом target можно в произвольном месте разметки, например так:

<div class="inner-part" title="Вложенный парт" data-state="init"></div>
{{render innerPart target=".inner-part"}}

или даже так:

{{render innerPart target=".inner-part"}}
<div class="inner-part" title="Вложенный парт" data-state="init"></div>

Результат будет тот же.

Таким образом, вместо создания вложенного элемента можно использовать уже существующий в разметке элемент с любым набором атрибутов.

Push-id, peek-id, pop-id

Однако, указание селектора требует придумывания новых ID или css-классов для целевых элементов. Причем они должны быть уникальны. В больших шаблонах это может быть проблемой. Чтобы решить ее введен специальный Handlebars-хелпер push-id и псевдо-селекторы :pop-id и :peek-id. Проще всего пояснить на примере:

<div id="{{push-id}}" title="Вложенный парт" data-state="init"></div>
{{render innerPart target=":pop-id"}}

Хелпер push-id просто генерирует новый уникальный ID (с помощью того же метода View.newId()), запоминает его и подставляет в текст шаблона. Псевдо-селектор :pop-id получает запомненный ID и используется его для ссылки на целевой элемент. При этом запомненный ID удаляется. Т.е., например, подобный шаблон приведет к ошибке:

<button id="{{push-id}}"></button>
{{data-bind text="btnText" target=":pop-id"}}
{{command-bind btnCommand target=":pop-id"}}

Хелпер command-bind выбросит исключение, так как запомненный ID будет удален предыдущим хелперов data-bind. Чтобы иметь возможность использовать ID несколько раз, следует применять псевдо-селектор :peek-id, который не удаляет запомненный ранее ID:

<button id="{{push-id}}"></button>
{{data-bind text="btnText" target=":peek-id"}}
{{command-bind btnCommand target=":pop-id"}}

Лучше не использовать :peek-id без необходимости: это позволит избежать ошибок.

Несмотря на глаголы push, pop и peek, традиционно используемые для стека, запоминается только последний ID, сгенерированный хелпером push-id. Выбор таких глаголов обусловлен схожей семантикой: в одном случае ID удаляется, а в другом нет.

Использование push-id и :pop-id/:peek-id работает немного быстрее, чем использование кастомных селекторов. Дело в том, что генерируемый ID уникальный, а это позволяет для поика элемента использовать супер-быстрый метод document.getElementById. Даже если используется селектор вида #id, поиск не такой быстрый, так как в этом случае нельзя гарантировать уникальность и область поиска ограничивается DOM-элементом, переданным в метод View.render. Как ни странно, но такой поиск работает медленнее, чем поиск во всем документе. Более сложные селекторы работают еще медленнее.
