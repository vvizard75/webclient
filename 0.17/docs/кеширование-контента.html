Кеширование контента

При разработке полезно отключить любое кеширование всего статического контента. Основная проблема - это кешироваине в браузере. Т.к. сервер (IIS) автоматически инвалидирует кеш при изменении файлов. Минимальный набор http-заголовков, которые сервер должен добавлять в ответ на запрос контекта, чтобы запретить браузеру кешировать контент:

Cache-Control: no-cache, no-store, must-revalidate
Pragma: no-cache
Expires: 0

Однако, задать в IIS'е настройками такую функциональность сложно. Самое просто, это задать "Cache-Control: no-cache". Для этого в web.config надо добавить:

<system.webServer>
        <staticContent>
            <clientCache cacheControlMode="DisableCache" />
        </staticContent>
    </system.webServer>

Важно иметь в виду, что некоторые прокси (например http://proxysgc.croc.ru) игнорируют "Cache-Control: no-cache", хотя данное значение явно запрещает им В частности кроковский прокси BlueCoat SG заменяет заголовок "Cache-Control: no-cache" на "Expires" с датой на 1.5 часа в прошлом. Что вообще говоря не то же самое. Поэтому, проще при разработке на стенд ходить напрямую, минуя прокси.

При внедрении приложения в бой, очевидно кеширование статического контента наоборот должно быть включено. Т.к. не смотря на то, что сервер кеширует контент и возвращает 304 (Not Modified), если в запросе были переданы заголовки "If-Modified-Since" и/или " If-None-Match" (и файл не менялся), это все равно дополнительный запрос. Кеширование для браузера управляется заголовками "Cache-Control: max-age=" и "Expires: ". При этом max-age имеет более высокий приоритет.

Однако, кеширование контента не должно мешать обновлению приложения. Т.к. если браузер закеширует контент, то ничто не может заставит его инвалидировать этот кеш. Единственный способ - это изменить url контента. Поэтому период для max-age/Expires должен быть разумный.
